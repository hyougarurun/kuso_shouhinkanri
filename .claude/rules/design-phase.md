# 設計フェーズルール

設計フェーズはTDDワークフローのSTEP 1。ここの品質がアプリ・システムの価値に直結する。

## 原則: 曖昧な状態で実装に進まない

- ユーザーの要件を100%理解するまで設計フェーズを完了しない
- 「たぶんこうだろう」「一般的にはこう」で推測しない
- 不明点は全てユーザーに質問する

## 質問の形式

### YES/NO質問（2択）
```
パスワードに大文字小文字の混在を必須にしますか？ → Y / N
```

### 選択式質問（3択以上）
```
認証方式を選択してください:
  A) メール+パスワードのみ
  B) メール+パスワード + OAuth（Google）
  C) メール+パスワード + OAuth（Google, GitHub）
  D) その他（具体的に教えてください）
  → A / B / C / D
```

### 数値質問
```
パスワードの最低文字数は何文字にしますか？ → 数値で回答
```

## 質問の網羅性

以下の観点を漏れなく確認する:

### 機能要件
- 正常系: 何をしたら何が起きるか
- 異常系: エラー時にどう振る舞うか
- 境界値: 上限・下限・空・最大長

### UI/UX
- 画面遷移: どのページからどのページに遷移するか
- 表示要素: 何を表示するか、表示順は
- インタラクション: クリック・入力・ドラッグ等の操作

### データ
- 入力データ: 何を受け取るか、バリデーションルール
- 出力データ: 何を返すか、フォーマット
- 永続化: DBに保存するか、何を保存するか

### 影響範囲
- 既存機能への影響はあるか
- 既存テストケースの変更が必要か
- 既存のAPIやDBスキーマの変更が必要か

## 質問の進め方

1. まず機能の全体像を把握する質問（3-5問）
2. 次に詳細を詰める質問（機能要件・UI/UX・データ）
3. 最後に影響範囲の確認

質問は一度にまとめて出す（1問ずつ聞かない）。ユーザーの負担を最小化する。

## 設計結果の保存

質問への回答が全て揃ったら:
1. 設計メモを `docs/design-notes/{機能名}.md` に保存する
2. 設計メモには以下を含める:
   - 機能概要
   - 確定した要件（質問への回答をまとめたもの）
   - 影響範囲
   - 実装方針（Phase分割案）
3. ユーザーに設計メモを確認してもらう
4. 承認後、STEP 2（テスト設計）に進む

## 禁止事項

- 質問せずに仮定を置いて進めること
- 「一般的にはこうする」で要件を埋めること
- 設計メモを保存せずにテスト設計に進むこと
- ユーザーの回答を待たずに実装を始めること
