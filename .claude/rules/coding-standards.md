# コーディング規約

## 基本原則

- 動くコードより読めるコードを優先する
- 1つの関数は1つの責務だけ持つ
- 推測でライブラリのAPIを使うな。公式ドキュメントを確認してから使え
- 既存コードのパターンに合わせろ。新しいパターンを勝手に導入するな

## 命名規則

- 変数・関数: camelCase
- 定数: UPPER_SNAKE_CASE
- クラス・型: PascalCase
- ファイル名: kebab-case（コンポーネントファイルはPascalCase可）
- テストファイル: `{対象}.test.{ext}` または `test_{対象}.{ext}`

## コメント

- コードの「何をしているか」ではなく「なぜそうしているか」を書く
- 自明なコードにコメントは不要
- TODO/FIXME/HACKを使う場合は理由と対応時期を併記する:
  ```
  // TODO(2026-05): ページネーション実装後に削除する一時的な全件取得
  ```

## エラーハンドリング

- エラーを握りつぶすな。catchしたら適切に処理するかre-throwしろ
- ユーザー向けエラーメッセージは日本語で、原因と対処法を含める
- システム境界（API呼び出し、DB操作、ファイルI/O）では必ずエラーハンドリングする
- 内部ロジックの過剰な防御的プログラミングは不要

## セキュリティ

- ユーザー入力は必ずバリデーション・サニタイズする
- SQLインジェクション: パラメータバインドを使う。文字列結合禁止
- XSS: ユーザー入力をHTMLに埋め込む際はエスケープする
- 認証・認可チェックはAPI層で必ず行う
- シークレット情報はコードにハードコードしない。環境変数を使う

## パフォーマンス

- N+1クエリを書くな。必要なデータは1クエリでまとめて取得する
- 大量データの処理はページネーション・ストリーミングを使う
- 不要な再レンダリングを避ける（React: useMemo, useCallback を適切に使う）

## 変更のスコープ

- 依頼された変更だけを行う。周辺コードの「ついでにリファクタ」は禁止
- 既存のコメント・型定義・テストを勝手に削除しない
- 未使用コードの削除は、ユーザーに確認してから行う
