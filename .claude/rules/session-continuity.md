# セッション継続ルール

Claude Codeはセッションをまたいで記憶を持たない。このルールでセッション間の作業継続を実現する。

## docs/progress.md の役割

`docs/progress.md` はプロジェクトの「作業引継ぎ書」。
新しいセッションが始まったとき、このファイルを読むだけで前回の続きから再開できる状態を維持する。

## セッション開始時（必須）

新しいセッションが始まったら、以下を自動で実行する:

1. `docs/progress.md` を読む
2. `docs/test-spec.md` のサマリーセクションを読む
3. `git log --oneline -10` で直近の変更を確認する
4. ユーザーに要約を報告する:
   ```
   前回の作業状況:
   - 機能: {機能名}
   - 進捗: Phase {X}/{Y} 完了
   - 次にやること: {次のタスク}
   - ブロッカー: {あれば記載}
   
   続きから再開しますか？ → Y / N
   ```

## docs/progress.md の更新タイミング

以下のタイミングで自動更新する:

1. **設計フェーズ完了時** — 確定した要件と影響範囲を記録
2. **テスト設計commit時** — 追加・更新したテストケースIDを記録
3. **各Phase完了時** — 完了したPhaseと次のPhaseを記録
4. **テスト実行後** — PASS/FAIL結果を記録
5. **セッション終了前** — 現在の状態を最終更新

## docs/progress.md のフォーマット

```markdown
# 作業進捗

## 現在の作業
- 機能: {機能名}
- 状態: {設計中 / テスト設計中 / Phase X/Y 実装中 / テスト中 / 完了}
- 次にやること: {具体的なタスク}
- ブロッカー: {あればここに。なければ「なし」}
- 未解決の設計判断: {あればここに。なければ「なし」}

## テスト状況
- TC-XXX-001: ✅ PASS
- TC-XXX-002: ✅ PASS  
- TC-XXX-003: ⬜ 未実装
- TC-XXX-004: ❌ FAIL（原因: {概要}）

## 直近の完了タスク
- {日付} Phase 1: {内容} (commit: {hash})
- {日付} Phase 2: {内容} (commit: {hash})

## 設計メモへのリンク
- docs/design-notes/{機能名}.md
```

## 注意事項

- progress.md はgit管理する（チームメンバーも状況を把握できるようにする）
- 1つの機能の作業が完了したら、progress.md の「現在の作業」をクリアして「直近の完了タスク」に移動する
- 複数機能を並行で進める場合は「現在の作業」セクションを複数作る
