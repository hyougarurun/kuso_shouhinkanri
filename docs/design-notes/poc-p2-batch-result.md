## 所見・発見事項

### 1. 学習データの全体像
- メガメガくん業者の請求は**月¥100〜180万円、明細 130〜180 件**が標準レンジ
- 全 12 ファイル合計で **1,814 件の明細** → 統計集計（PoC-P3）の母集団として十分な量

### 2. 別業者（202501/202502 系）
- 明細わずか 2 件 → おそらく月額固定の継続サービス（Base の出店料・決済手数料？）
- 学習対象外と明示し、`data/parsed/*.json` の読み込み時に `taxAmount === 0 && lineItems.length <= 3` で除外する

### 3. パーサー改善ポイント（PoC-P3 以前に修正推奨）
- **`ParsedInvoice` 型に `issuer` / `invoiceNumber` の optional フィールドを追加**（現在は捨てている）
- 現状の `.raw.txt` には `issuer` / `invoiceNumber` がすでに抽出されているので、parser 側の型追加だけで即活用可能
- ただし k2 確認時のとおり、メガメガくん請求書は `issuer` 自体が PDF に明記されていない可能性が高い（null になる見込み）

### 4. パーサーの精度
- 明細カウントが **2026/01 の期待値 167 件** とほぼ一致（手動数えと突合要だが、8 ページのPDFにメガメガ 14-1 front などの項目がびっしり詰まっていることと整合）
- 全 14 ファイルで stop_reason=end_turn（truncate 0 件） → max_tokens=32k Stream API は妥当
